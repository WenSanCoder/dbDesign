# Redis 与 RabbitMQ 中间件介绍及课设应用方案

## 1. 文档目的

本项目计划设计一个“高校智慧教务与校园管理系统”，其中“高并发选课”是系统的重要亮点。选课场景天然存在瞬时并发高、热门课程名额竞争激烈、不能超选、不能重复选课等问题。如果所有请求都直接访问数据库，数据库会承受较大压力，也更容易出现锁竞争和响应变慢。

因此，本项目考虑引入两个常见企业级中间件：

- Redis：用于热点数据缓存、课程容量预扣减、限流、分布式锁或原子操作。
- RabbitMQ：用于选课请求削峰填谷、异步排队处理、提升系统抗瞬时流量能力。

这两个技术在课设中的定位不是为了“炫技”，而是围绕一个明确业务问题：

```text
如何在大量学生同时抢热门课程时，保证系统响应快、数据库压力可控，并且最终不会出现课程超选和重复选课？
```

## 2. Redis 简介

### 2.1 Redis 是什么

Redis 是一个基于内存的高性能键值数据库，常用于缓存、计数器、排行榜、分布式锁、限流、会话存储等场景。

Redis 的特点：

- 数据主要存储在内存中，读写速度很快。
- 支持多种数据结构，不只是简单字符串。
- 支持原子操作，适合做库存扣减、计数、限流等。
- 支持过期时间，适合缓存临时数据。
- 支持 Lua 脚本，可以把多个操作组合成一个原子操作。
- 可以通过持久化机制将内存数据保存到磁盘。

在本项目中，Redis 主要用于解决“选课开始瞬间大量学生同时查询和扣减课程余量”的问题。

### 2.2 Redis 常见数据结构

#### String 字符串

最常用的数据结构，可以存储字符串、数字、JSON 字符串等。

本项目可用于：

- 存储教学班剩余容量。
- 存储验证码。
- 存储登录 Token 黑名单。
- 存储简单计数器。

示例：

```text
course:capacity:1001 -> 30
```

表示教学班 1001 当前 Redis 中剩余容量为 30。

#### Hash 哈希

适合存储对象。

本项目可用于：

```text
teaching_class:1001
  courseName -> 数据库系统
  teacherName -> 张老师
  capacity -> 60
  selectedCount -> 30
```

#### List 列表

适合简单队列，但企业项目中如果需要更可靠的消息队列，一般会使用 RabbitMQ、Kafka 等专门消息队列。

#### Set 集合

适合存储不重复元素。

本项目可用于：

- 记录某学生已选择的教学班。
- 记录某教学班已选学生集合。

示例：

```text
student:selected:20240001 -> {1001, 1002, 1003}
```

#### Sorted Set 有序集合

适合排行榜、带分数排序的数据。

本项目可用于：

- 候补队列。
- 热门课程排名。
- 课程评价排名。

示例：

```text
course:hot_rank
  teachingClassId=1001 score=1500
  teachingClassId=1002 score=1200
```

### 2.3 Redis 为什么适合高并发选课

高并发选课中有一个核心数据：课程剩余容量。

如果每个学生选课都直接查数据库：

```text
查询教学班剩余容量 -> 判断是否有余量 -> 插入选课记录 -> 更新已选人数
```

在热门课程高并发情况下，会出现：

- 数据库请求突然暴涨。
- 同一行教学班数据被大量事务竞争更新。
- 锁等待增加。
- 接口响应变慢。
- 如果并发控制不好，可能出现超选。

Redis 的优势是：

- 读写快，适合保存热点课程容量。
- 单线程命令执行模型保证单条命令天然原子。
- `DECR`、`INCR` 等操作适合计数。
- Lua 脚本可以把“判断容量是否大于 0”和“扣减容量”合成一个原子操作。

### 2.4 Redis 在本项目中的定位

在本项目中，Redis 不作为最终数据库，而是作为高并发选课的“前置缓冲层”。

它负责：

- 缓存教学班剩余容量。
- 快速判断热门课程是否还有名额。
- 原子预扣减课程容量。
- 减少大量请求直接打到 openGauss。
- 可选：做接口限流，防止单个学生频繁点击选课。

它不负责：

- 保存最终选课结果。
- 作为成绩、学生、课程等核心数据的主数据库。
- 替代 openGauss 的事务和约束。

最终选课结果必须落到 openGauss 中，openGauss 才是最终可信数据源。

## 3. RabbitMQ 简介

### 3.1 RabbitMQ 是什么

RabbitMQ 是一个消息队列中间件。它的核心作用是让系统之间通过“消息”进行异步通信。

简单理解：

```text
生产者发送消息 -> RabbitMQ 保存消息 -> 消费者取出消息处理
```

在高并发系统中，消息队列常用于：

- 异步处理。
- 流量削峰。
- 系统解耦。
- 任务排队。
- 失败重试。
- 延迟处理。

在本项目中，RabbitMQ 主要用于“学生提交选课请求后，不立即同步落库，而是先进入队列，由后端消费者按顺序或按能力处理”。

### 3.2 RabbitMQ 核心概念

#### Producer 生产者

发送消息的一方。

在本项目中，Spring Boot 选课接口就是生产者。学生提交选课请求后，后端将请求封装成消息发送到 RabbitMQ。

#### Consumer 消费者

接收并处理消息的一方。

在本项目中，选课处理服务就是消费者。它从 RabbitMQ 中取出选课请求，执行数据库事务，最终生成选课结果。

#### Queue 队列

消息真正排队等待消费的地方。

本项目可以定义一个选课队列：

```text
course.selection.queue
```

#### Exchange 交换机

生产者通常不是直接把消息发到队列，而是发给交换机，再由交换机根据规则路由到队列。

常见交换机类型：

- Direct Exchange：根据路由键精确匹配。
- Fanout Exchange：广播到所有绑定队列。
- Topic Exchange：根据通配符匹配路由键。

本项目用 Direct Exchange 就足够。

#### Routing Key 路由键

用于决定消息进入哪个队列。

本项目可以使用：

```text
course.selection
```

#### Message 消息

消息是生产者发送给队列的数据。

本项目中的选课消息可以包含：

```json
{
  "requestId": "REQ202607010001",
  "studentId": 10001,
  "teachingClassId": 20001,
  "roundId": 1,
  "requestTime": "2026-07-01 10:00:00"
}
```

### 3.3 RabbitMQ 为什么适合高并发选课

选课系统的难点在于瞬时流量。

假设选课开始一瞬间有 3000 个学生同时点击热门课程，如果所有请求都同步进入数据库：

- 数据库连接池可能被打满。
- 教学班容量行会发生严重锁竞争。
- 用户等待时间变长。
- 系统可能出现接口超时。

RabbitMQ 的作用是把瞬时流量变成可控的排队流量：

```text
3000 个请求瞬间进入系统
  -> 快速写入 RabbitMQ
  -> 后端消费者按照处理能力逐步消费
  -> 数据库压力稳定在可控范围
```

它的核心价值是：

- 削峰：把瞬时高峰变成平稳处理。
- 异步：用户请求不必一直阻塞等待数据库事务完成。
- 解耦：选课接口和选课落库逻辑分离。
- 重试：消费者处理失败后可以重试或记录失败。

### 3.4 RabbitMQ 在本项目中的定位

RabbitMQ 在本项目中负责：

- 接收选课请求消息。
- 对选课请求排队。
- 控制消费者处理速度。
- 避免数据库被瞬时流量打爆。
- 支持失败重试或失败记录。

RabbitMQ 不负责：

- 判断选课是否一定成功。
- 保证课程容量一定不超卖。
- 替代数据库事务。

选课是否成功最终仍由 openGauss 中的事务、约束和业务校验决定。

## 4. Redis 与 RabbitMQ 的区别

Redis 和 RabbitMQ 都可以出现在高并发架构中，但它们解决的问题不同。

| 对比项 | Redis | RabbitMQ |
|---|---|---|
| 核心定位 | 高性能缓存/内存数据结构 | 消息队列 |
| 主要解决问题 | 热点数据读写、计数、限流、原子扣减 | 异步处理、削峰填谷、系统解耦 |
| 数据形态 | Key-Value、Hash、Set、ZSet 等 | Message |
| 本项目用途 | 缓存课程余量，原子预扣减 | 选课请求排队，异步落库 |
| 是否保存最终业务数据 | 否 | 否 |
| 是否替代数据库 | 否 | 否 |
| 与 openGauss 关系 | 减少热点查询和扣减压力 | 平滑写入数据库的请求压力 |

一句话总结：

```text
Redis 负责“快判断、快扣减”，RabbitMQ 负责“排队慢慢处理”，openGauss 负责“最终正确”。
```

## 5. 课设中的整体应用方案

### 5.1 选课业务问题建模

本项目中的选课业务需要满足以下规则：

- 学生只能在选课轮次开放时间内选课。
- 学生不能重复选择同一个教学班。
- 学生不能选择时间冲突的课程。
- 学生不能选择未满足先修课要求的课程。
- 教学班已选人数不能超过容量。
- 热门课程高并发下不能出现超选。
- 用户应尽快得到请求受理结果。
- 最终选课结果必须和数据库一致。

### 5.2 整体架构图

```text
学生浏览器
   |
   | 1. 点击选课
   v
Vue 前端
   |
   | 2. 调用选课 API
   v
Spring Boot 后端
   |
   | 3. 基础校验
   v
Redis
   |
   | 4. Lua 脚本原子预扣减课程余量
   v
RabbitMQ
   |
   | 5. 选课请求进入队列
   v
选课消费者服务
   |
   | 6. openGauss 事务落库
   v
openGauss 数据库
   |
   | 7. 保存最终选课结果
   v
前端轮询/通知查询结果
```

### 5.3 详细流程

#### 第一步：管理员发布选课轮次

管理员在系统中配置：

- 学期。
- 选课轮次。
- 选课开始时间。
- 选课结束时间。
- 可选课程范围。
- 教学班容量。

这些数据最终保存到 openGauss。

#### 第二步：预热课程容量到 Redis

选课开始前，系统从 openGauss 查询教学班容量和已选人数：

```text
剩余容量 = 教学班容量 - 已选人数
```

然后写入 Redis：

```text
course:capacity:{teachingClassId} = 剩余容量
```

示例：

```text
course:capacity:20001 = 50
course:capacity:20002 = 30
```

#### 第三步：学生提交选课请求

学生点击“选课”按钮，前端向后端发送请求：

```json
{
  "teachingClassId": 20001,
  "roundId": 1
}
```

后端从 Token 中获取学生身份，不允许前端直接传 studentId 作为可信身份。

#### 第四步：后端基础校验

后端先做不依赖高并发扣减的基础校验：

- 用户是否登录。
- 用户是否是学生。
- 选课轮次是否开放。
- 教学班是否存在。
- 教学班是否允许当前学生选择。
- 学生是否已经提交过相同选课请求。

这一层校验能提前拦截明显非法请求。

#### 第五步：Redis 原子预扣减

后端使用 Redis Lua 脚本执行：

1. 判断课程剩余容量是否存在。
2. 判断剩余容量是否大于 0。
3. 如果大于 0，则扣减 1。
4. 返回扣减成功或失败。

伪代码：

```lua
local key = KEYS[1]
local stock = tonumber(redis.call('get', key))

if stock == nil then
    return -1
end

if stock <= 0 then
    return 0
end

redis.call('decr', key)
return 1
```

返回含义：

```text
1  -> 预扣减成功
0  -> 没有剩余容量
-1 -> Redis 中不存在该课程容量
```

如果 Redis 返回没有容量，接口可以直接告诉学生：

```text
课程名额已满
```

如果 Redis 预扣减成功，说明请求获得了一个“临时名额”，继续进入 RabbitMQ。

#### 第六步：发送选课消息到 RabbitMQ

后端构造选课消息：

```json
{
  "requestId": "REQ202607010001",
  "studentId": 10001,
  "teachingClassId": 20001,
  "roundId": 1,
  "requestTime": "2026-07-01 10:00:00"
}
```

发送到：

```text
Exchange: course.selection.exchange
RoutingKey: course.selection
Queue: course.selection.queue
```

接口快速返回：

```json
{
  "status": "PROCESSING",
  "message": "选课请求已提交，正在处理中"
}
```

这样用户请求不会长时间阻塞在数据库事务上。

#### 第七步：消费者异步处理选课请求

选课消费者从 RabbitMQ 中取出消息，然后执行数据库事务。

事务内需要再次校验：

- 学生是否存在。
- 教学班是否存在。
- 选课轮次是否有效。
- 是否重复选课。
- 是否有时间冲突。
- 是否满足先修课。
- 教学班容量是否仍然可用。

数据库层仍然需要最终兜底，因为 Redis 只是预扣减，不是最终事实。

#### 第八步：openGauss 事务落库

事务处理逻辑：

1. 查询教学班当前已选人数和容量。
2. 判断容量是否充足。
3. 判断是否已有相同选课记录。
4. 判断课表是否冲突。
5. 插入选课记录。
6. 更新教学班已选人数。
7. 写入操作日志。
8. 提交事务。

如果成功：

```text
student_course_selection.status = selected
```

如果失败：

```text
student_course_selection.status = failed
fail_reason = 具体原因
```

同时需要回补 Redis 容量：

```text
INCR course:capacity:{teachingClassId}
```

### 5.4 数据一致性设计

本系统必须强调：Redis 和 RabbitMQ 提升性能，但不能替代数据库一致性。

数据库层需要设计以下约束：

#### 防止重复选课

在选课记录表建立唯一约束：

```sql
UNIQUE (student_id, teaching_class_id)
```

即使两个请求同时进入数据库，也只能成功一条。

#### 防止课程超选

更新教学班已选人数时使用条件更新：

```sql
UPDATE teaching_class
SET selected_count = selected_count + 1
WHERE teaching_class_id = ?
  AND selected_count < capacity;
```

如果影响行数为 0，说明容量不足，选课失败。

#### 保证事务原子性

插入选课记录和更新教学班人数必须在同一个事务中完成。

如果任何一步失败，事务回滚。

#### 记录失败原因

失败原因需要保存，方便学生查看：

```text
课程容量已满
时间冲突
重复选课
未满足先修课要求
选课轮次已结束
```

### 5.5 前端交互设计

学生点击选课后，不建议直接显示“选课成功”，而是显示：

```text
选课请求已提交，正在处理中
```

前端可以通过两种方式获取最终结果：

#### 方式一：轮询

每隔 1-2 秒查询一次选课结果：

```text
GET /api/student/selections/result?requestId=REQ202607010001
```

优点：

- 实现简单。
- 适合课设。

#### 方式二：WebSocket

后端处理完后主动推送结果。

优点：

- 实时性更好。

缺点：

- 实现复杂度更高。
- 课设时间紧时不建议作为必做。

本项目建议优先使用轮询。

## 6. 后端接口设计建议

### 6.1 预热选课容量

接口：

```text
POST /api/admin/selection-rounds/{roundId}/preheat
```

作用：

- 从 openGauss 查询教学班剩余容量。
- 写入 Redis。
- 用于选课开始前准备。

### 6.2 学生提交选课

接口：

```text
POST /api/student/selections
```

请求：

```json
{
  "teachingClassId": 20001,
  "roundId": 1
}
```

响应：

```json
{
  "requestId": "REQ202607010001",
  "status": "PROCESSING",
  "message": "选课请求已提交，正在处理中"
}
```

### 6.3 查询选课结果

接口：

```text
GET /api/student/selections/result/{requestId}
```

响应：

```json
{
  "requestId": "REQ202607010001",
  "status": "SELECTED",
  "message": "选课成功"
}
```

失败示例：

```json
{
  "requestId": "REQ202607010001",
  "status": "FAILED",
  "message": "课程容量已满"
}
```

### 6.4 学生退课

接口：

```text
POST /api/student/selections/{selectionId}/drop
```

退课成功后：

- 更新数据库选课状态。
- 教学班已选人数减 1。
- Redis 中课程剩余容量加 1。
- 写入操作日志。

## 7. 数据库表设计补充

为了配合 Redis 和 RabbitMQ，数据库中建议增加一些字段。

### 7.1 student_course_selection 选课记录表

建议字段：

| 字段 | 含义 |
|---|---|
| selection_id | 选课记录编号 |
| request_id | 请求编号 |
| student_id | 学生编号 |
| teaching_class_id | 教学班编号 |
| round_id | 选课轮次 |
| status | 选课状态 |
| fail_reason | 失败原因 |
| selected_at | 选课时间 |
| dropped_at | 退课时间 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

其中 `request_id` 用于前端查询异步处理结果。

### 7.2 selection_request_log 选课请求日志表

可以额外设计一个选课请求日志表，用于记录 RabbitMQ 消息处理状态。

| 字段 | 含义 |
|---|---|
| request_id | 请求编号 |
| student_id | 学生编号 |
| teaching_class_id | 教学班编号 |
| request_status | 请求状态 |
| mq_status | 消息状态 |
| retry_count | 重试次数 |
| error_message | 错误信息 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

该表不是必须，但对报告和答辩很有帮助，因为它能说明异步任务的可追踪性。

## 8. 课设实现版本规划

考虑课设时间有限，可以分为三个版本推进。

### 8.1 保底版

只使用 openGauss 完成同步选课。

特点：

- 实现简单。
- 数据库事务完整。
- 能保证不超选、不重复选课。

实现方式：

- 使用唯一约束防重复。
- 使用事务保证插入选课记录和更新人数一致。
- 使用条件更新防止超选。

适合在 Redis/RabbitMQ 来不及完成时兜底。

### 8.2 标准亮点版

使用 Redis + openGauss。

特点：

- Redis 缓存容量。
- Redis Lua 预扣减。
- openGauss 最终落库。
- 不引入 RabbitMQ，接口同步处理数据库事务。

优点：

- 比保底版更有高并发亮点。
- 实现难度中等。

缺点：

- 没有完整削峰排队能力。

### 8.3 完整亮点版

使用 Redis + RabbitMQ + openGauss。

特点：

- Redis 原子预扣减。
- RabbitMQ 异步排队。
- openGauss 事务落库。
- 前端轮询结果。
- 日志表记录处理状态。

这是本项目最推荐的目标版本。

## 9. 答辩讲解话术

答辩时可以这样讲：

```text
我们的系统将高并发选课作为核心业务场景。传统做法是所有学生请求直接访问数据库，这会导致热门课程的容量行出现大量锁竞争，数据库压力集中，用户响应变慢。

因此我们设计了 Redis + RabbitMQ + openGauss 的架构。

Redis 用于缓存教学班剩余容量，并通过 Lua 脚本完成原子预扣减，快速拦截没有名额的请求。

RabbitMQ 用于接收已经通过 Redis 预扣减的选课请求，把瞬时高峰流量转化为后端消费者可控的异步处理流量。

openGauss 是最终数据源，负责通过事务、唯一约束和条件更新保证最终一致性，防止重复选课和课程超选。

也就是说，Redis 解决速度问题，RabbitMQ 解决削峰问题，openGauss 解决最终正确性问题。
```

如果老师问“为什么不用 Redis 直接保存选课结果”，可以回答：

```text
Redis 是缓存和中间件，不适合作为本系统最终业务数据源。选课记录、成绩、学生信息等核心数据必须保存在 openGauss 中，才能满足关系约束、事务一致性、统计查询和课程设计要求。
```

如果老师问“RabbitMQ 是否能保证不超选”，可以回答：

```text
RabbitMQ 本身不负责判断是否超选，它只是让请求排队并削峰。真正防止超选的是数据库事务、条件更新和唯一约束。我们把数据库作为最后一道一致性保障。
```

## 10. 项目中的最终推荐方案

本课设最终推荐采用：

```text
Redis + RabbitMQ + openGauss
```

具体分工：

- Redis：课程容量缓存、原子预扣减、可选限流。
- RabbitMQ：选课请求排队、异步削峰。
- openGauss：最终选课记录、事务一致性、约束校验、统计查询。
- Spring Boot：业务接口、消息生产者、消息消费者。
- Vue：学生选课页面、处理状态展示、结果轮询。

最终目标不是证明 Redis 或 RabbitMQ 多复杂，而是证明我们能根据业务场景合理选择中间件，并且知道中间件与数据库之间的职责边界。

核心原则：

```text
中间件提升性能和可用性，
数据库保证最终一致性和可审计性。
```
