# 高校智慧教务与校园管理系统项目总体方案

## 1. 课程设计任务要求梳理

### 1.1 课程设计总体目标

本次数据库系统课程设计要求以关系数据库应用系统设计与实现为核心，围绕一个具体的信息系统完成从需求分析、概念结构设计、逻辑结构设计、物理结构设计、数据库实现到应用系统开发与运行的完整过程。

根据课程任务书，本项目原则上应以 openGauss DBMS 作为实验和实践平台，通过标准 SQL 语言完成数据库实现，并在此基础上选择一种程序设计语言或开发框架完成应用系统。课程重点不只是做出一个能运行的 Web 系统，而是要体现数据库系统设计全过程，包括：

- 对用户需求进行分析，整理数据需求、功能需求、处理需求和性能需求。
- 建立数据库概念模型，绘制 E-R 图，包括实体、属性和实体间联系。
- 将 E-R 图转换为关系模式，完成逻辑结构设计和关系模式优化。
- 完成物理结构设计，包括索引、视图、分区、备份策略等。
- 使用 SQL 创建数据库、基本表、完整性约束、索引、视图、触发器和存储过程。
- 开发一个可运行的应用系统，并在报告中说明开发环境、系统功能、运行截图和数据库连接方式。
- 提交完整实验报告、源程序以及所有 SQL 语句文件。

### 1.2 硬性数据库要求

根据任务书、课堂补充要求和系统功能自查清单，本项目必须满足以下数据库相关要求：

- 原则上使用 openGauss 作为数据库管理系统。
- 自选信息系统最终至少包含 6 个以上基本表，且表之间必须存在主外键关联。
- 数据库、基本表、字段命名需要满足个性化要求。
- 创建基本表时需要考虑并建立恰当的数据完整性约束机制。
- 需要建立用户视图，用于响应用户查询和统计需求。
- 需要建立索引，提高关键查询和统计性能。
- 至少创建 2 个触发器，用于完成特殊数据更新或维护功能。
- 至少创建 2 个存储过程，用于完成特殊查询、统计或批处理功能。
- 数据库、基本表、索引、视图等创建成功后，需要将 DBMS 环境下含 SQL 命令的运行窗口截图放入实验报告。
- 基本表数据插入、修改、删除命令执行成功后，需要将运行窗口截图放入实验报告。
- 查询或统计命令的执行结果也需要截图，用于验证 SQL 语句与查询结果一致。
- 最终需要单独提交一个包含全部 SQL 语句的 `.sql` 文件。

### 1.3 个性化命名要求

课程要求每个小组成员根据小组组号和姓名建立各自的个性化数据库。

数据库命名方式：

```text
姓的全拼 + 名的第一个拼音 + MIS + 小组组号
```

示例：

```text
JiangrgMIS01
```

基本表命名方式：

```text
姓的全拼 + 名的第一个拼音 + _ + 汉字或英文单词 + 小组组号
```

示例：

```text
Jiangrg_Students01
```

属性命名方式：

```text
姓名的第 1 个拼音字母 + _ + 汉字或英文单词 + 小组组号
```

示例：

```text
jrg_Sno01
jrg_Sname01
```

因此，本项目在团队统一数据库设计的基础上，每个成员最终需要生成一份符合自己姓名和组号规则的 SQL 文件，并在自己的数据库中完成系统运行和截图验证。

## 2. 项目选题与总体定位

### 2.1 项目名称

项目名称建议定为：

```text
高校智慧教务与校园管理系统
```

或：

```text
高校智慧教务管理系统
```

如果希望更突出课程设计亮点，可以在报告中使用完整名称：

```text
面向高并发选课场景的高校智慧教务与校园管理系统
```

### 2.2 项目定位

本项目面向高校教务管理场景，围绕学生、教师、教务管理员三个核心角色，完成学籍管理、课程管理、教学班管理、选课管理、成绩管理、统计分析和智能选课辅助等功能。

系统不是简单的学生成绩管理 CRUD，而是以数据库设计为核心，结合企业级 Web 应用开发方式，构建一个具有真实业务闭环的教务平台 Demo。项目重点体现：

- 数据库概念结构、逻辑结构和物理结构设计能力。
- openGauss 数据库对象设计能力。
- Web 前后端系统开发能力。
- 高并发选课场景下的架构设计能力。
- 基于需求调研驱动系统设计的产品分析能力。
- 基于 openGauss DataVec 或 GaussDB 向量能力的 AI 选课辅助创新能力。

### 2.3 项目核心业务闭环

系统核心业务闭环如下：

1. 管理员维护学院、专业、班级、学生、教师、课程、教学班等基础数据。
2. 教务管理员配置学期、选课轮次、课程容量、课程时间、课程限制。
3. 学生在选课开放时间内查询可选课程，进行选课和退课。
4. 系统检查容量、时间冲突、重复选课、先修课、年级或专业限制。
5. 高并发选课请求通过 Redis 和 RabbitMQ 进行削峰和异步处理。
6. 教师查看自己负责的教学班和学生名单。
7. 教师录入成绩并提交。
8. 系统通过约束、触发器和事务保证成绩范围、成绩变更审计和学分自动汇总。
9. 学生查询课表、成绩、已修学分和通知。
10. 管理员和教师查看成绩统计、课程统计、生源统计、选课统计等报表。
11. 智能选课助手基于课程介绍、教师评价、历史成绩和学生需求提供课程推荐。

## 3. 需求分析设计方案

### 3.1 需求调研目标

本项目计划进行轻量但完整的需求调研。调研不追求样本规模很大，而是重点体现需求分析过程真实、逻辑合理，并且能够影响后续数据库和系统功能设计。

调研目标包括：

- 了解学生在选课、课表、成绩、学分、教师评价方面的真实痛点。
- 了解教师在成绩录入、学生名单查看、课程统计方面的需求。
- 了解教务管理员在基础数据维护、教学班安排、选课管理和统计分析方面的需求。
- 提炼系统的功能性需求。
- 提炼系统的非功能性需求，例如并发能力、数据一致性、安全性、易用性和可维护性。
- 将调研结论映射到数据库表设计、业务流程设计和系统模块设计中。

### 3.2 调研对象

建议调研对象包括：

- 本校或同专业学生 15-30 人。
- 任课教师、助教或班主任 1-2 人。
- 具有教务系统使用经验的同学若干。

如果时间有限，可以采用问卷为主、访谈为辅的方式。

### 3.3 学生问卷方向

学生问卷围绕以下方向设计：

#### 选课体验

- 是否遇到过选课系统卡顿、排队、无法提交等问题。
- 是否遇到过热门课程名额很快被抢完的问题。
- 是否希望看到课程剩余容量实时变化。
- 是否需要候补机制。
- 是否希望系统自动检查时间冲突。

#### 课程信息透明度

- 选课时是否关注教师评分、课程难度、作业量、考试方式。
- 是否希望查看课程历史平均分、及格率、优秀率。
- 是否希望查看同学对课程和教师的评价。
- 是否希望根据兴趣方向推荐课程。

#### 成绩和学分

- 是否经常查询已修学分。
- 是否希望系统自动提示毕业要求差距。
- 是否关注课程绩点、排名、平均分等统计信息。

#### 通知与易用性

- 是否希望选课结果、退课结果、成绩发布通过系统通知。
- 是否希望移动端也能方便访问。
- 对现有教务系统界面和操作流程有哪些不满意。

### 3.4 教师访谈方向

教师或助教访谈围绕以下方向设计：

- 教师是否需要快速查看自己本学期所有教学班。
- 教师是否需要导出学生名单。
- 成绩录入时最担心的问题是什么，例如录错、漏录、重复提交。
- 是否需要查看课程平均分、最高分、最低分、成绩分布。
- 成绩提交后是否需要审核或保留修改记录。
- 是否希望系统对成绩异常情况进行提醒。

### 3.5 教务管理员访谈方向

教务管理员需求可通过模拟访谈或场景分析获得，重点包括：

- 学院、专业、班级、学生、教师等基础数据如何维护。
- 每学期课程、教学班、上课时间和容量如何配置。
- 如何控制选课轮次、选课时间和选课范围。
- 是否需要查看课程选课人数、满课率、退课率。
- 是否需要按地区、学院、专业统计学生信息。
- 是否需要生成教学班、成绩、学分等统计报表。

### 3.6 先射箭后画靶的调研映射

为了服务项目最终设计，调研问题需要围绕我们已经规划的数据库模式和系统模块进行设计。这样报告中能够形成清晰的“调研发现 -> 需求提炼 -> 数据库设计 -> 功能实现”闭环。

| 调研方向 | 预期发现 | 对应需求 | 对应数据库设计 |
|---|---|---|---|
| 选课卡顿与热门课程抢课 | 热门课程存在高并发访问和容量竞争 | 高并发选课、容量控制、排队处理 | teaching_class、student_course_selection、course_selection_round |
| 时间冲突 | 学生担心选到时间冲突课程 | 自动检测课表冲突 | class_schedule、student_course_selection |
| 课程信息不透明 | 学生希望看到教师评价、课程难度和历史成绩 | 课程评价、教师评分、历史成绩统计 | course_review、teacher、grade_record |
| 学分进度不清晰 | 学生希望知道已修学分和毕业差距 | 自动统计已修学分 | grade_record、credit_summary |
| 成绩录入易出错 | 教师担心成绩范围错误、修改无记录 | 成绩约束、成绩修改审计 | grade_record、operation_log |
| 教务统计工作繁琐 | 管理员需要快速统计选课和成绩 | 统计视图、存储过程、报表页面 | view、procedure、selection_log |
| 模糊选课需求 | 学生会用自然语言描述课程偏好 | AI 选课助手、语义检索 | course_knowledge、teacher_review_vector |

### 3.7 功能性需求

系统功能性需求包括：

- 用户登录与身份认证。
- 管理员、教师、学生多角色权限控制。
- 学院、专业、班级管理。
- 学生信息管理。
- 教师信息管理。
- 课程信息管理。
- 教学班信息管理。
- 学期和选课轮次管理。
- 学生选课、退课、候补和结果查询。
- 课表查询。
- 教师任课查询。
- 教师成绩录入和提交。
- 学生成绩查询。
- 学分自动统计。
- 学生成绩排名。
- 课程平均成绩统计。
- 生源地统计。
- 班级课程开设查询。
- 操作日志和成绩修改审计。
- AI 选课辅助查询。

### 3.8 非功能性需求

系统非功能性需求包括：

- 并发性：热门课程选课时支持大量学生同时提交请求。
- 一致性：不能出现课程超选、重复选课、成绩学分不一致等问题。
- 安全性：不同角色只能访问自己的权限范围。
- 可维护性：后端分层清晰，数据库结构规范，SQL 可重复执行。
- 可扩展性：后续可以扩展培养方案、毕业审核、教室管理等功能。
- 可审计性：关键操作需要记录日志，成绩修改需要可追踪。
- 可用性：界面清晰，常见操作路径短，适合验收演示。
- 可部署性：支持本地运行和云服务器部署。

## 4. 系统模块设计

### 4.1 用户与权限模块

该模块负责系统登录、角色管理和权限控制。

主要功能：

- 用户登录。
- JWT Token 鉴权。
- 用户密码加密存储。
- 管理员、教师、学生三类角色划分。
- 菜单权限控制。
- 接口权限控制。

主要角色：

- 系统管理员：维护全局数据、用户、角色、课程和统计报表。
- 教务管理员：维护教务基础数据、教学班、选课轮次。
- 教师：查看任课、查看学生名单、录入成绩。
- 学生：选课、退课、查课表、查成绩、查看学分。

### 4.2 基础数据管理模块

该模块负责学校基础组织结构和人员数据维护。

主要功能：

- 学院管理。
- 专业管理。
- 班级管理。
- 地区管理。
- 学生管理。
- 教师管理。
- 学期管理。

数据库重点：

- 学院与专业是一对多关系。
- 专业与班级是一对多关系。
- 班级与学生是一对多关系。
- 地区与学生是一对多关系。
- 教师归属于学院。

### 4.3 课程与教学班模块

该模块负责课程和每学期开课安排。

主要功能：

- 课程信息维护。
- 课程学分、学时、考核方式维护。
- 课程先修关系维护。
- 教学班创建。
- 教学班容量设置。
- 教学班任课教师设置。
- 教学班上课时间和地点维护。
- 班级课程开设查询。

数据库重点：

- 课程是稳定的课程目录。
- 教学班是某学期某课程的一次具体开课。
- 一个课程在同一学期可以开设多个教学班。
- 一个教师一学期可以教授多个教学班。
- 一个教学班可以设置多条上课时间。

### 4.4 选课模块

该模块是系统核心业务模块，也是并发架构亮点所在。

主要功能：

- 查看可选课程。
- 按课程名、教师、学分、时间、学院筛选课程。
- 查看课程剩余容量。
- 学生选课。
- 学生退课。
- 候补排队。
- 查看选课结果。
- 选课冲突检测。
- 重复选课检测。
- 先修课检测。
- 专业或年级限制检测。

数据库重点：

- 选课记录表记录学生和教学班的选课关系。
- 同一学生不能重复选择同一教学班。
- 教学班容量不能超出上限。
- 学生不能选择时间冲突的教学班。
- 选课状态需要区分处理中、已选、候补、退课、失败。

### 4.5 成绩与学分模块

该模块负责教师成绩录入和学生成绩查询。

主要功能：

- 教师查看教学班学生名单。
- 教师录入成绩。
- 教师修改成绩。
- 教师提交成绩。
- 管理员审核成绩。
- 学生查询成绩。
- 学生查询已修学分。
- 成绩排名。
- 每门课程平均分统计。
- 学年成绩统计。

数据库重点：

- 成绩必须限制在 0-100 分之间。
- 成绩修改需要记录日志。
- 成绩达到及格线后自动计入已修学分。
- 学分汇总可以通过触发器或存储过程维护。

### 4.6 统计分析模块

该模块用于展示数据库查询、视图、存储过程的能力。

主要功能：

- 按课程统计平均分、最高分、最低分、及格率。
- 按学生统计学年成绩和排名。
- 按学生统计已修课程和已修学分。
- 按教师查询任课信息。
- 按班级查询课程开设情况。
- 按地区统计学生人数。
- 统计选课人数、满课率、退课率。

数据库重点：

- 使用视图简化复杂查询。
- 使用存储过程封装常见统计逻辑。
- 使用索引优化高频筛选字段。

### 4.7 通知与审计模块

该模块体现系统的企业级规范。

主要功能：

- 选课成功通知。
- 退课成功通知。
- 成绩发布通知。
- 成绩变更通知。
- 登录日志。
- 选课操作日志。
- 成绩修改日志。

数据库重点：

- 日志表可以按时间分区。
- 关键业务表修改时通过触发器写入审计记录。

### 4.8 AI 选课助手模块

该模块作为创新功能，不作为系统主线，但可以作为高分亮点。

主要功能：

- 学生输入自然语言需求。
- 系统理解课程偏好，例如“给分友好”“偏实践”“人工智能方向”“作业少”。
- 系统结合课程简介、教师评价、历史成绩和结构化条件推荐课程。
- 返回推荐课程列表和推荐理由。

示例问题：

```text
帮我推荐评分高、作业量适中、不要和周三上午冲突的人工智能方向课程。
```

```text
我还差 4 个专业选修学分，有哪些比较适合大三学生的课？
```

```text
帮我找历史平均分较高、老师评价好的数据库相关课程。
```

实现层级可以分为两档：

- 保底版：基于结构化 SQL 和关键词匹配返回推荐结果。
- 提升版：基于 openGauss DataVec 或 GaussDB 向量检索实现语义召回，再结合 SQL 条件过滤。

## 6. 应用分析

### 6.1 为什么选择高校教务系统

高校教务系统天然适合数据库课程设计，原因包括：

- 实体丰富：学生、教师、学院、专业、班级、课程、教学班、成绩、选课记录等。
- 关系明确：一对多、多对多关系都较典型，适合 E-R 建模。
- 查询统计多：成绩排名、平均分、学分统计、生源地统计、教师任课查询等都能体现 SQL 能力。
- 完整性要求高：成绩范围、选课容量、重复选课、主外键关联都需要数据库约束。
- 并发场景真实：热门课程选课存在高并发请求和容量竞争。
- 可扩展性好：可以扩展通知、评价、推荐、审计、AI 助手等功能。

### 6.2 系统用户分析

#### 学生用户

学生主要关注：

- 课程是否能选上。
- 课程是否和已有课表冲突。
- 课程容量是否充足。
- 任课教师评价如何。
- 历史成绩分布如何。
- 自己成绩和学分是否满足要求。

#### 教师用户

教师主要关注：

- 查看自己任课课程。
- 查看教学班学生名单。
- 录入和修改成绩。
- 查看课程成绩统计。
- 保证成绩提交过程准确、可追溯。

#### 教务管理员

教务管理员主要关注：

- 维护基础数据。
- 安排课程和教学班。
- 设置选课轮次。
- 监控选课情况。
- 统计学生、课程、成绩和教学资源。
- 管理异常数据。

### 6.3 核心业务场景

#### 场景一：学生高并发选课

学生在选课开始后同时选择热门课程。系统需要保证：

- 页面能快速响应。
- 课程容量不能超卖。
- 学生不能重复选课。
- 失败原因清晰可查。
- 选课结果最终和数据库一致。

#### 场景二：教师成绩录入

教师完成课程考试后录入成绩。系统需要保证：

- 成绩只能在合法范围内。
- 成绩修改需要留下审计记录。
- 成绩提交后学生可以查询。
- 及格课程自动计入学生已修学分。

#### 场景三：管理员统计分析

教务管理员需要查询某课程平均分、某教师任课情况、某地区学生人数、某班级开课情况等。系统需要通过视图、索引和存储过程提高统计效率。

#### 场景四：AI 选课咨询

学生用自然语言描述选课偏好，系统结合课程文本信息、历史成绩和教师评价给出推荐。该场景体现 openGauss 向量检索能力和数据库扩展应用能力。

## 9. 数据库对象设计计划

### 9.1 视图设计

计划至少创建以下视图：

- 学生课表视图：整合学生、课程、教学班、教师、上课时间。
- 教师任课视图：整合教师、课程、教学班、学期、选课人数。
- 学生成绩视图：整合学生、课程、成绩、学分、绩点。
- 课程成绩统计视图：统计平均分、最高分、最低分、及格率。
- 生源地统计视图：按地区统计学生数量。

### 9.2 索引设计

计划创建以下索引：

- 学生学号唯一索引。
- 教师工号唯一索引。
- 课程代码唯一索引。
- 选课记录 `(student_id, teaching_class_id)` 唯一索引。
- 选课记录 `teaching_class_id` 普通索引。
- 成绩记录 `student_id` 普通索引。

### 9.3 触发器设计

用途：

用途：

- 体现成绩管理的可追溯性。
- 防止成绩被无记录修改。
- 支持报告中的企业级规范说明。

可选触发器：

- 选课成功后自动生成通知。
- 选课记录状态变化时更新教学班已选人数。

## 10. 高并发选课架构设计

### 10.1 业务问题

高校选课存在典型高并发问题：

- 热门课程开始选课瞬间请求集中。
- 多名学生同时抢最后几个名额。
- 普通数据库直接扣减容量可能导致锁竞争。
- 如果缺少约束，可能出现重复选课或课程超选。
- 用户体验上不能长时间等待。

### 10.2 架构方案

高并发选课建议采用：

```text
前端请求
  -> Spring Boot 接口
  -> Redis 校验和容量预扣减
  -> RabbitMQ 排队
  -> 消费者异步处理
  -> openGauss 事务落库
  -> 返回最终选课结果
```

### 10.3 处理流程

1. 管理员发布选课轮次。
2. 系统将教学班容量预热到 Redis。
3. 学生提交选课请求。
4. 后端先校验登录状态、学生身份、选课时间、课程状态。
5. Redis 使用 Lua 脚本原子判断和扣减剩余容量。
6. 扣减成功后，将选课请求发送到 RabbitMQ。
7. 接口快速返回“选课处理中”。
8. RabbitMQ 消费者异步处理选课请求。
9. 消费者开启 openGauss 数据库事务。
10. 数据库再次校验是否重复选课、是否容量超限、是否时间冲突。
11. 校验成功后插入选课记录并更新教学班已选人数。
12. 校验失败则记录失败原因，并回补 Redis 容量。
13. 学生前端轮询选课结果，或通过通知模块查看最终状态。

### 10.4 一致性保障

本系统需要强调 Redis 和 RabbitMQ 不是替代数据库一致性，而是提升系统吞吐和削峰能力。最终一致性由 openGauss 保证。

一致性手段包括：

- Redis Lua 脚本保证容量预扣减原子性。
- RabbitMQ 保证请求排队处理。
- openGauss 事务保证落库原子性。
- `student_id + teaching_class_id` 唯一约束防止重复选课。
- 条件更新防止教学班容量超卖。
- 操作日志记录失败原因和处理结果。

- DataVec 向量表。
- HNSW 或 IVFFLAT 向量索引。
- 语义检索 TopK。
- 结合 SQL 过滤选课条件。
- 简易 AI 对话窗口。

该亮点既能体现数据库新技术探索，也能服务需求调研中“课程信息不透明、学生希望智能推荐”的痛点。
